# Промпт для исполнения (соло-режим)

Работаем по Ральф-методологии. Ты — автономный агент в **solo-режиме**, выполняющий задачи из `docs/tasks/tasks.json` **без оркестратора, без reviewer и без test coordinator**.

## Жёсткий режим SOLO (антипутаница)

- Считай, что этот промпт имеет приоритет над любыми оркестраторными инструкциями в `agent_instructions` внутри `tasks.json`.
- Ты работаешь один и закрываешь задачу end-to-end в одной сессии.
- В этом режиме **запрещены** статусы `needs_review`, `blocked`, `failed` как рабочий финал цикла.
- Единственный рабочий путь статусов: `pending -> work in progress -> done`.
- Поля координации оркестратора (`assignee`, `claimed_at`, `lease_until`, `review_required`, `test_verdict`, `test_owner`, `test_started_at`, `test_finished_at`, `artifacts`, `status_reason`) не являются обязательными шагами solo-цикла:
  - не заполняй их «для процесса»;
  - не жди внешнего review/test-verdict;
  - не переводи задачу в `needs_review`.
- Если в задаче/файле встречаются оркестраторные указания, трактуй их как нерелевантные для solo-режима и продолжай по этому документу.

## Инициализация

1. Прочитай `docs/new-agents.md` (правила проекта).
2. Прочитай `docs/tasks/tasks.json` (список задач).
3. Выполни в **корне репозитория** (workspace root) команду: `git log --oneline -20` — чтобы увидеть последние коммиты; шаг не пропускать, выполнять через терминал.
4. Найди задачу для работы:
   - статус: `pending`;
   - все `dependencies` имеют статус `done`;
   - наивысший приоритет (`critical > high > medium > low`).

## Цикл работы над задачей

1. **Объяви задачу**: «Беру TASK-XXX: <описание>».
2. **Обнови `docs/tasks/tasks.json`**: измени `status` на `"work in progress"`.
3. **Реализуй** `acceptance_criteria`.
4. **Проверь (обязательно, до смены `status` на `done`)**:
   - Если `package.json` уже есть: `npm install` при отсутствующем `node_modules`, затем `npm run compile` и `npm test`.
   - Если текущая задача создаёт TypeScript VS Code extension scaffold и `package.json` ещё нет: сначала создай scaffold, затем выполни команды из добавленных `package scripts`.
   - Если задача только документационная: выполни `test_steps`, проверь ссылки/пути и не выдумывай npm-команды без `package.json`.
   - Не помечай задачу выполненной и не делай commit, пока применимый quality gate не выполнен успешно.
   - Выполни `test_steps` из задачи (ручные шаги — опиши результат или «требует ручной проверки»).
5. **Обнови `docs/tasks/tasks.json`**: измени `status` на `"done"` *только после* успешного применимого quality gate (`npm run compile`, `npm test`, `test_steps`).  
   *(Исключение: отсутствующий scaffold до TASK-001 или security/privacy-тесты, явно отложенные в pending задачах текущей очереди.)*
6. **Запиши прогресс** в `docs/tasks/progress.md` (1-2 строки: что сделано).
7. **Сделай git commit** с осмысленным сообщением.

### Что делать, если задача «просит review»

- Если формулировки в задаче подразумевают reviewer/test coordinator, в solo-режиме это заменяется твоей собственной полной проверкой (`npm run compile` + полный доступный `npm test` + `test_steps`).
- После этой проверки сразу переводи задачу в `done` (без промежуточного `needs_review`).

## После завершения задачи

Напиши:

```text
TASK-XXX выполнена (N/M). Следующая pending-задача: TASK-YYY (<описание>).
```

Где:

- `N` — число задач в статусе `done`;
- `M` — общее число задач в `tasks.json`.

## Правила

- Работай только над **одной** задачей за раз.
- Не трогай код, не связанный с текущей задачей.
- Не меняй и не удаляй задачи — только разрешённые поля (в первую очередь `status`).
- Если задача требует чего-то, чего нет в `dependencies` — остановись, сообщи и зафиксируй проблему в `docs/tasks/currentProblems.md`.
- При ошибках в тестах:
  - исправь регрессии, если изменения сломали проект;
  - обнови автотесты, если это требуется после изменения кода и не ухудшает качество проверки;
  - не переходи к следующей задаче, пока не закроешь текущий цикл.

## Когда все задачи `done`

- Перенеси `docs/tasks/tasks.json` в `docs/tasks/done` (через команды shell).
- Создай в `docs/releaseNotes` файл `.md`: `YYYY-MM-DD.md` или `YYYY-MM-DD-краткий-смысл.md`; при коллизии имени — числовой суффикс.
- Релиз-нот: отрази пользовательские изменения по задачам батча для расширения VS Code; без внутренних технических деталей и идентификаторов задач в пользовательском тексте.

Приступай к исполнению и не останавливайся, пока не закроешь текущий цикл по задаче до `done`.
