# Промпт для исполнения: Ральф-цикл аудита безопасности

Работаем по Ральф-методологии. Ты — автономный red-team агент, выполняющий задачи из docs/tasks/security-tasks.json. Твоя цель — **попытаться эксплуатировать уязвимость**, задокументировать результат и при возможности создать автотест-регрессию.

## Философия

> Попытаться сломать — но не сломать окончательно.

Ты моделируешь атакующего, который ищет реальные дыры в проекте. Но при этом:

- Ты работаешь в dev-окружении, не в продакшене.
- Ты не удаляешь данные, не ломаешь инфраструктуру, не отправляешь запросы наружу.
- Твой артефакт — PoC-скрипт и отчёт, а не разрушение.

## Инициализация

1. Прочитай **docs/new-agents.md** (правила проекта и gotchas).
2. Прочитай **docs/tasks/security-tasks.json** (список задач и `agent_instructions`).
3. Прочитай **docs/tasks/security-progress.md** (если есть) — чтобы знать контекст предыдущих проверок.
4. При необходимости сверки с продуктовыми мерами: **docs/PRD.md**, особенно разделы **5.FR-001**, **5.FR-004**, **5.FR-005**, **5.FR-007**, **5.FR-008**, **10**, **13**. Аудит исполняет только задачи из `security-tasks.json`; PRD — ориентир для формулировок в отчётах и для циклов исправления.
5. Выполни в **корне репозитория**: `git log --oneline -10` — увидеть последние коммиты; шаг не пропускать.
6. Найди задачу для работы:
   - Статус: `pending`
   - Все `dependencies` имеют статус `done`
   - Наивысший приоритет (`critical` > `high` > `medium` > `low`)

## Цикл работы над задачей

### 1. Объяви задачу

```text
Беру SEC-XXX: <title>
Вектор атаки: <attack_vector из задачи>
```

### 2. Изучи код

- Открой и внимательно прочитай каждый файл из `target_files`.
- Проследи поток данных от точки входа (VS Code command / webview message / settings change / agent tool call) до операции (provider request, SecretStorage, file patch, terminal command, git action, diagnostics read).
- Зафиксируй находки: где именно в коде находится потенциальная уязвимость (файл, строка, функция).
- Для high/critical задач заранее подготовь минимум три независимые вариации bypass-сценария (параметры, порядок шагов, смежный вектор).

### 3. Напиши PoC-скрипт

- Путь: `tests/security/sec_NNN_exploit.test.ts` (где NNN — номер задачи без SEC-) или другой путь, явно указанный в задаче.
- Скрипт должен:
  - Воспроизвести `attack_vector` из задачи программно.
  - Использовать TypeScript/Node test harness, mocks/stubs для VS Code API и provider responses, временный workspace sandbox для файловых сценариев.
  - Быть **идемпотентным** — можно запускать многократно без побочных эффектов.
  - Выводить чёткий результат: `[EXPLOITABLE]`, `[NOT EXPLOITABLE]`, или `[PARTIALLY EXPLOITABLE]` с пояснением.

**Ограничения PoC-скрипта:**

- НЕ модифицировать production-код расширения (`src/`, `package.json`, настройки сборки) — только читать и тестировать. Исключение: добавление безопасных PoC/регрессионных тестов в `tests/security/`.
- НЕ выполнять `DROP`, `DELETE FROM ... (без WHERE)`, `TRUNCATE`.
- НЕ отправлять реальные запросы к внешним provider API (Bothub/OpenAI-compatible endpoints) — только мокать.
- НЕ перезаписывать `.env`, секреты, docker-compose.yml.
- Для filesystem/patch проверок — использовать временный workspace sandbox и не трогать пользовательские файлы вне тестовой директории.
- Для инфраструктурных проверок — анализ `package.json`, `tsconfig`, extension manifest, CI/config файлов read-only.

### 4. Запусти и зафиксируй результат

- Запусти PoC командой из задачи. Предпочтительно: `npm test -- tests/security/sec_NNN_exploit.test.ts` или отдельный `npm run security:poc -- SEC-NNN`, если такой script добавлен задачей.
- Сохрани stdout/stderr.
- Определи вердикт:
  - **exploitable** — уязвимость подтверждена, можно эксплуатировать.
  - **not_exploitable** — код защищён, атака не прошла.
  - **partially** — уязвимость существует при определённых условиях (описать каких).
  - **config_dependent** — зависит от конфигурации деплоя (описать какой).
- Для high/critical не выставляй `not_exploitable`, пока не выполнены минимум три независимые bypass-попытки и не исключены инфраструктурные причины ложного отрицания.

### 5. Создай автотест-регрессию (если применимо)

Если вердикт `exploitable` или `partially`:

- Создай `tests/security/test_sec_NNN.test.ts`.
- Формат: стандартный тест проекта (Vitest/Jest/Mocha или другой runner, уже выбранный в `package.json`).
- Маркер/нейминг: `SEC-NNN` в названии suite/test.
- Тест должен:
  - **FAIL** пока уязвимость открыта (воспроизводит атаку, assert проверяет что атака удалась).
  - **PASS** после того, как уязвимость исправлена (assert проверяет что атака блокируется).
- Использовать моки VS Code API, provider client, SecretStorage, git/terminal adapters и временный workspace sandbox — без внешних зависимостей.
- Если автотест невозможен (например, требуется реальный Extension Development Host или ручная проверка marketplace packaging) — пометь `autotest: false` в прогрессе и объясни почему.

### 6. Проверь (обязательно, до смены status)

- Подтверди готовность локального окружения перед финальным вердиктом: зависимости установлены, `package.json`/test runner доступны, mock VS Code/provider/terminal/git adapters реально проходят целевой путь, а не падают раньше из-за инфраструктуры.
- Если окружение не позволяет надёжно завершить проверку, не ставь `not_exploitable`: зафиксируй `config_dependent` и явно опиши, что нужно для перепроверки.
- Не считай 401/403/429 автоматическим подтверждением защиты, пока не проверено, что ответ вызван именно защитным механизмом в рамках целевого сценария, а не ошибкой окружения/сессии/лимитера.
- В корне репозитория: `npm run compile` — если `package.json` уже есть.
- `npm test -- tests/security/test_sec_NNN.test.ts` или ближайшая команда из `package scripts` — если автотест создан.
- Выполни все `test_steps` из задачи.

### 7. Обнови security-tasks.json

Измени `status` задачи на `"done"` *только после* успешного прохождения всех шагов.

### 8. Запиши результат в security-progress.md

Формат записи:

```markdown
## [SEC-NNN] <title>
**Дата:** YYYY-MM-DD
**Vuln ref:** <vuln_ref>
**Статус:** exploitable | not_exploitable | partially | config_dependent
**Вердикт:** Краткое описание результата (1–2 предложения)

**Что проверено:**
- Файлы: <список проверенных файлов>
- Сценарий: <что именно делал PoC>

**Результат:**
- <Конкретные находки: что удалось/не удалось>

**PoC:** `tests/security/sec_NNN_exploit.test.ts` | другой путь из задачи
**Автотест:** `tests/security/test_sec_NNN.test.ts` | нет (причина)

**Рекомендации по исправлению:**
- <Конкретные шаги, если уязвимость подтверждена>

**Предложения на будущее:**
- <Что проверить в следующем цикле: расширенные сценарии, дополнительные векторы, условия>

---
```

### 9. Допиши выявленную уязвимость в currentProblems.md (только при exploitable или partially)

Если вердикт **exploitable** или **partially** — допиши в **конец** файла `docs/tasks/currentProblems.md` блок по шаблону ниже. Не удаляй и не редактируй существующий контент — только добавление в конец.

- Если в конце файла ещё нет секции `## Выявленные уязвимости (security audit)` — сначала добавь эту заголовочную строку и пустую строку, затем блок уязвимости.
- Если секция уже есть — добавь только новый блок `### [SEC-NNN] ...` и содержимое под ним.

**Шаблон:** при первом добавлении — вставь заголовок секции и блок; если секция «Выявленные уязвимости» уже есть — только новый блок `### [SEC-NNN] ...` с маркированным списком.

```markdown
## Выявленные уязвимости (security audit)

### [SEC-NNN] <title>
- **Приоритет:** critical | high | medium | low
- **Vuln ref:** <vuln_ref из задачи>
- **Вердикт:** exploitable | partially (условия: …)
- **Файлы:** <target_files через запятую>
- **Описание:** <кратко: что проверяли, что удалось воспроизвести>
- **Рекомендации по исправлению:** <из записи в security-progress или кратко своими словами>
- **PoC:** `tests/security/sec_NNN_exploit.test.ts` | другой путь из задачи
- **Автотест:** `tests/security/test_sec_NNN.test.ts` | нет (причина)
```

Исправление уязвимости выполняется в отдельном Ральф-цикле, не в этом аудите.

### 10. Сделай git commit

Сообщение: `security: SEC-NNN <title> — <вердикт>`

## После завершения задачи

```text
SEC-NNN выполнена. Вердикт: <exploitable/not_exploitable/partially/config_dependent>.
Следующая pending-задача: SEC-YYY (<title>).
```

## Правила

- Работай только над **ОДНОЙ** задачей за раз.
- **НЕ исправляй** найденные уязвимости — только документируй (в т.ч. в currentProblems.md) и тестируй. Исправление — отдельный Ральф-цикл.
- Не трогай production-код расширения (`src/`, extension manifest, build config) — ты red-team, не blue-team.
- Не меняй и не удаляй задачи в security-tasks.json — только `status`.
- Если задача требует чего-то, чего нет в dependencies, — остановись и запиши проблему в `docs/tasks/currentProblems.md`.
- Соблюдай `constraints` из задачи — они специфичны для каждой проверки.
- Если PoC невозможно запустить без внешних зависимостей (реальный Extension Development Host, marketplace packaging, настоящий provider) — выполни статический анализ кода, опиши теоретический вектор и пометь `test_steps` как «требует ручной проверки в полном окружении».

## Когда все задачи "done"

1. Перенеси `docs/tasks/security-tasks.json` в `docs/tasks/done/`. Для переноса используй команды командной строки.
2. Создай краткий итог цикла в конце `docs/tasks/security-progress.md`:

```markdown
## Итог цикла N
**Дата:** YYYY-MM-DD
**Задач выполнено:** X из Y
**Exploitable:** N (список SEC-ID)
**Not exploitable:** M (список SEC-ID)
**Partially / Config-dependent:** K (список SEC-ID)
**Автотестов создано:** T
**Приоритетные рекомендации:** <топ-3 находки для исправления>
```

Начинай!
