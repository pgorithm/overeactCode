# Промпт для генерации списка задач на рефакторинг PRD Overeact Code

Ты — инициализирующий агент для рефакторинга документа требований. Твоя задача — по **полному содержимому docs/PRD.md** и по **фактической реализации продукта** сформировать структурированный список **задач на рефакторинг PRD** в формате JSON.

**Обязательный охват:**

1. **Весь PRD целиком** — Overview, Target Audience, MVP Scope, User Flows, Functional Requirements, Data Model, UX, Platform, Integrations, Security/Privacy, Milestones, Risks, Assumptions, Open Questions. Искать противоречия **между любыми** частями документа.
2. **MVP канон** — согласовать разделы 3–12 с текущей очередью `docs/tasks/tasks.json`: side composer, provider config, agent session, retrieval-first context, patch proposals, permission policies, verification loop, diagnostics, git awareness и dogfooding.
3. **Согласование с реализацией** — там, где PRD утверждает поведение, настройки, команды, модели данных, UI-flow или интеграции, которых в коде нет (или наоборот), итоговый канон PRD должен **соответствовать текущему коду и конфигурации**, если иное явно не зафиксировано как roadmap/open question. Для сверки используй **репозиторий**, **тесты**, `docs/tasks/tasks.json` и **docs/tasks/progress.md**.

**Основная цель:** сделать PRD единой картиной продукта Overeact Code, а не смесью roadmap, задач и устаревших предположений. Параллельно: убрать или переписать требования, которые **устарели**, **противоречат** друг другу или **опровергнуты** реализацией/очередью задач; не тащить в канон отменённые или заменённые решения и дубликаты.

Выполнение рефакторинга разбито на **несколько задач**, чтобы каждая задача умещалась в контекстное окно одного агента.

## Контекст процесса

1. **docs/PRD.md** описывает MVP VS Code extension. По мере реализации могут расходиться требования, задачи, open questions и фактический код.
2. **Цель рефакторинга:** привести **весь** PRD к **одной непротиворечивой** картине. Приоритет истины: **текущая реализация и progress.md** → `docs/tasks/tasks.json` → актуальные решения в PRD → старые предположения/open questions. Сохраняй полезный контекст (граничные случаи, приёмка), но **удаляй** требования, которые отменены, заменены или дублируют уже описанный канон.
3. **Этот** промпт создаёт **docs/tasks/prd_refactor_tasks.json** — список задач на такой рефакторинг. Каждая задача выполняется отдельным агентом за одну сессию (работа только с документом PRD, без изменения кода).
4. Референс по формату задач: [docs/prompts/generateTasksAdditions.md](docs/prompts/generateTasksAdditions.md) — корневой объект с `agent_instructions` и `tasks`; у задач рефакторинга **дополнительно** поля `target_section` и `sections_to_review`. Задачи здесь — не на реализацию в коде, а на планомерное **редактирование PRD** (слияние дубликатов, выбрасывание устаревшего, обновление ссылок, нумерации и формулировок).

## Входные данные

1. **docs/PRD.md** — **целиком** (обязательно для генерации списка задач): все разделы 1–15, заголовки и перекрёстные ссылки.
2. **docs/tasks/progress.md** — чтобы не описывать в PRD как «нужно сделать» то, что уже закрыто, и чтобы фиксировать принятые решения.
3. **docs/tasks/tasks.json** — текущая очередь MVP; используй как мост между PRD и реализацией.
4. **Код и конфигурация** — выборочно, по темам расхождений (VS Code extension entrypoint, package scripts, webview, provider client, SecretStorage, policy evaluator, patch/terminal/git/diagnostics adapters). Цель — устранить расхождения «PRD ↔ факт».
5. **Тесты** (при споре о поведении) — как вторичный источник ожидаемого поведения после кода.

## Выходной файл

- **Путь**: **docs/tasks/prd_refactor_tasks.json** (создай или перезапиши этот файл).
- **Содержимое**: только задачи на рефакторинг PRD. Идентификаторы задач — **PRD-REFACTOR-001**, **PRD-REFACTOR-002**, … (отдельная нумерация от TASK-xxx в основном tasks.json). При необходимости задачи из этого файла можно объединить с docs/tasks/tasks.json вручную или скриптом.

## Требования к выходному файлу

### 1. Формат — строго JSON

Используй JSON, а не Markdown. Структура — как в [docs/prompts/generateTasksAdditions.md](docs/prompts/generateTasksAdditions.md): корневой объект с полями `agent_instructions` и `tasks` (массив задач), плюс у каждой задачи поля **`target_section`** и **`sections_to_review`** (см. пример ниже).

**`sections_to_review`:** список разделов PRD, которые нужно открыть для этой задачи (например `["5.FR-004", "6.PatchProposal", "10"]`). Если задача — сквозная вычитка на противоречия и соответствие коду, укажи несколько разделов и опиши это в `description` / `acceptance_criteria`.

```json
{
  "agent_instructions": {
    "before_start": [
      "Прочитай docs/PRD.md (целиком или разделы, указанные в задаче)",
      "Роль оркестратор: при нескольких независимых PRD-задачах — atomic claim на каждую и параллельные воркеры по docs/prompts/RALPH-CURSOR_ORCHESTRATOR.md.",
      "Роль воркер: одна задача pending (соло) или одна выданная в work in progress; наивысший приоритет.",
      "Проверь, что все dependencies этой задачи выполнены (status done)"
    ],
    "during_work": [
      "Работай только над выбранной задачей рефакторинга",
      "Редактируй только docs/PRD.md; не меняй код",
      "При правках основного текста сверяйся с кодом (при расхождениях), docs/tasks/tasks.json и docs/tasks/progress.md — **реализация**, **progress** и текущая очередь имеют приоритет над устаревшим текстом в PRD",
      "Удали или сожми устаревшие требования и дубликаты; не оставляй двух взаимоисключающих описаний одного и того же поведения",
      "Сохраняй согласованность нумерации, оглавления и перекрёстных ссылок внутри PRD"
    ],
    "before_finish": [
      "Выполни все test_steps из задачи",
      "Меняй status на done только после проверки",
      "Кратко опиши выполненное в docs/tasks/progress.md (опционально, если ведётся учёт рефакторинга)"
    ]
  },
  "tasks": [
    {
      "id": "PRD-REFACTOR-001",
      "category": "prd_refactor",
      "target_section": "5.FR-004",
      "priority": "high",
      "description": "[Рефакторинг PRD] Согласовать требования к patch workflow, diff preview и сохранению unrelated user changes",
      "sections_to_review": ["3", "5.FR-004", "6.PatchProposal", "7", "10", "13"],
      "acceptance_criteria": [
        "Требования к patch proposals, diff preview и применению изменений не противоречат друг другу",
        "Связь между FR-004, PatchProposal data model, UX и security/privacy описывает один и тот же workflow",
        "Если код уже реализует часть workflow иначе — PRD отражает факт или явно помечает отличие как roadmap/open question"
      ],
      "test_steps": [
        "Открыть docs/PRD.md и указанные sections_to_review",
        "Проверить, что нет двух разных описаний одного patch/diff workflow",
        "Проверить, что ссылки на задачи/разделы и формулировки acceptance criteria согласованы"
      ],
      "dependencies": [],
      "status": "pending"
    }
  ]
}
```

### 2. Принципы декомпозиции рефакторинга

**Одна задача — один смысловой фокус:** чаще всего один **целевой раздел** (например `5.FR-003` или `10`) и ограниченный набор связанных разделов; либо одна задача на **глобальную** вычитку (термины, ссылки, согласованность roadmap/open questions по всему файлу); либо задача «согласовать FR-001 с реализацией provider configuration».

Каждая задача должна быть выполнима за одну сессию агента при чтении указанных частей PRD (и при необходимости точечного чтения кода). Ориентир: 2–6 связанных разделов PRD на одну задачу; при большом объёме текста — меньше.

**Логическая группировка:** объединяй в одну задачу разделы, которые описывают одну подсистему: provider config/SecretStorage, side composer UI, retrieval/context, filesystem/patches, permission policy, terminal verification, diagnostics, git awareness, agent summaries, security/privacy, milestones/open questions.

**Зависимости:** если рефакторинг одного раздела зависит от стабилизации другого (например, security/privacy зависит от канона permission policy), укажи dependency от соответствующей задачи.

**Поле target_section:** укажи целевой раздел PRD (`3`, `5.FR-001`, `6.AgentSession`, `10`, `12` и т.д.). Это помогает агенту понять, какой фрагмент файла открыть первым.

**Поле sections_to_review:** как в подпункте «Формат» выше — список разделов PRD для сверки.

### 3. Категория и приоритет

- **category**: всегда **prd_refactor** для всех задач в этом файле.
- **priority**: **high** — рефакторинг разделов, от которых зависят задачи MVP или безопасность (provider config, permissions, patch workflow, verification); **medium** — UX/интеграции/milestones; **low** — косметика, терминология, объединение мелких пунктов.

### 4. Маппинг подсистем → разделы PRD

При анализе PRD используй такой ориентир:

- **Provider configuration:** разделы 4, 5.FR-001, 6.ProviderConfig, 6.ModelAssignment, 9, 10.
- **Side composer и UX:** разделы 3, 4, 5.FR-002, 5.FR-009, 7, 11.
- **Agent session/tool log:** разделы 4, 5.FR-009, 6.AgentSession, 6.ToolCallRecord.
- **Retrieval/context:** разделы 3, 5.FR-003, 6.ContextItem, 8, 13.
- **Filesystem/patch workflow:** разделы 5.FR-004, 6.PatchProposal, 7, 10, 13.
- **Terminal verification/diagnostics:** разделы 5.FR-005, 5.FR-006, 9, 12.
- **Git awareness:** разделы 5.FR-007, 9, 10, 13.
- **Permission/security/privacy:** разделы 5.FR-008, 6.PermissionPolicy, 10, 13.
- **Milestones/open questions:** разделы 12, 14, 15 и связанные FR.

### 5. Количество задач

Задач должно быть **не слишком много** (чтобы не дробить рефакторинг на десятки шагов) и **не слишком мало** (чтобы каждая задача была посильна одному агенту). Ориентир: **5–15 задач** на полный цикл (вычитка всего PRD + согласование с реализацией/очередью). Добавь 1–3 задачи на сквозные вещи (термины, ссылки, roadmap/open questions), если они нужны.

## Выполни

1. Прочитай **docs/PRD.md целиком**. Зафиксируй противоречия между разделами, устаревшие формулировки и расхождения с **текущей реализацией/очередью** (при сомнении — точечно открой код/тесты/progress/tasks).
2. Составь список подсистем PRD и связанных разделов для каждой.
3. Сгруппируй разделы в задачи так, чтобы в одной задаче был один смысловой фокус и объём текста был приемлем для одной сессии агента.
4. Добавь отдельные задачи на: **согласование ключевых FR с кодом** (где есть расхождения); **сквозную** правку терминов/ссылок/open questions; при необходимости — разделы milestones и risks. В **`target_section`** укажи основной затронутый раздел.
5. Для каждой задачи задай: id (PRD-REFACTOR-NNN), target_section, sections_to_review, description, acceptance_criteria, test_steps, dependencies, status.
6. Расставь приоритеты и зависимости (например, сначала глобальная терминология при необходимости, затем provider/security/patch/verification, затем UX/milestones/open questions).
7. Сгенерируй **docs/tasks/prd_refactor_tasks.json** с блоком **agent_instructions** и массивом **tasks**.

## Важно

- Рефакторинг не подразумевает изменения кода или тестов — только правки **docs/PRD.md** (для ответа на вопрос «как сейчас в продукте» **нужно** сверяться с кодом, тестами, `docs/tasks/tasks.json` и `docs/tasks/progress.md`; в PRD переносится уже согласованная формулировка).
- Результат — не механический перенос абзацев, а **единое описание того, как продукт устроен и ведёт себя на момент рефакторинга** (видимое поведение, границы системы, принятые решения), сверенное с кодом и тестами. Это **не** хронология правок PRD: отменённые или устаревшие ветки требований не сохраняем как действующие.
- По всему документу: **нет противоречий** между разделами; формулировки **либо** совпадают с реализацией, **либо** явно помечены как план/roadmap; в задачах рефакторинга **acceptance criteria** задаются только для того, что **остаётся обязательным требованием**.
- **Проверка согласованности:** если бы по обновлённому PRD с нуля реализовали продукт, получилось бы **то же поведение и те же границы ответственности**, что у текущего кода в репозитории.
- После рефакторинга PRD должен ясно отделять MVP, later features, assumptions и open questions. Не создавай искусственный раздел 3.5, если проектный канон уже живёт в разделах 1–15.

В **`acceptance_criteria`** и **`test_steps`** задач в `prd_refactor_tasks.json` явно включай проверки вида: «нет дублирующего противоречивого описания», «устаревшие формулировки в целевом разделе удалены или заменены», где это уместно.
