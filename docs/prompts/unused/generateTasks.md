# Промпт для трансформации docs/PRD.md -> docs/tasks/tasks.json

Ты — инициализирующий агент для RALPH-очереди задач. Твоя задача — преобразовать `docs/PRD.md` в структурированный `docs/tasks/tasks.json`, чтобы другие агенты могли выполнять проект инкрементально: одна атомарная задача за сессию или один worker на одну задачу под управлением оркестратора.

Не приступай к реализации продукта. Твоя работа ограничена анализом PRD и созданием файлов очереди задач.

## Цель

Создать первый полный backlog задач по PRD:

- задачи покрывают MVP и критический путь из `docs/PRD.md`;
- каждая задача атомарна, проверяема и имеет явные зависимости;
- файл соответствует канонической структуре `docs/tasks/tasks.template.json`;
- очередь пригодна для RALPH-оркестрации и solo-исполнения через отдельные промпты.

## Входные Данные

Перед генерацией прочитай:

1. `docs/new-agents.md` — обязательные правила и gotchas проекта.
2. `docs/PRD.md` — основной источник требований.
3. `docs/tasks/tasks.template.json` — канонический формат `tasks.json`.
4. Если существуют: `docs/tasks/progress.md`, `docs/tasks/currentProblems.md` — только для контекста и предотвращения дублирования.

Если `docs/PRD.md` отсутствует, не выдумывай требования. Сообщи, что нужен PRD, и остановись.

## Выходные Файлы

Создай или обнови:

- `docs/tasks/tasks.json` — основной результат, строго валидный JSON.
- `docs/tasks/progress.md` — создай, если файла ещё нет; если файл уже есть, не перезаписывай историю.

Не добавляй Markdown-комментарии внутрь `tasks.json`. JSON должен парситься стандартным парсером.

## Формат tasks.json

Используй структуру из `docs/tasks/tasks.template.json` как источник правды. Минимальный пример:

```json
{
  "agent_instructions": {
    "before_start": [
      "Прочитай этот файл, docs/new-agents.md и git log --oneline -20",
      "Роль оркестратор (команда /ralph или ведение очереди целиком): следуй docs/prompts/RALPH-CURSOR_ORCHESTRATOR.md и docs/RALPH-ORCHESTRATION-QUICKSTART.md — ready-задачи, 1..K без конфликтов, atomic claim на каждую, затем K параллельных воркеров (отдельные сессии/Task, одна задача на воркера); Test Coordinator и Reviewer по протоколу.",
      "Роль воркер: ровно одна задача в work in progress с твоим assignee; не бери вторую из очереди сам.",
      "Проверь, что все dependencies релевантной задачи имеют статус done"
    ],
    "during_work": [
      "Работай только над выбранной задачей",
      "Перед стартом задачи переведи status в work in progress и выстави assignee, claimed_at, lease_until",
      "После реализации переведи status в needs_review и добавь артефакты проверок"
    ],
    "before_finish": [
      "Запусти uv run ruff check . и task-focused тесты; full-suite pytest в parallel режиме подтверждает test-coordinator (или python -m ... fallback)",
      "Выполни ВСЕ test_steps из задачи",
      "Меняй status на done только после подтверждения reviewer"
    ]
  },
  "tasks": [
    {
      "id": "TASK-001",
      "category": "functional",
      "priority": "high",
      "description": "Краткое описание задачи на русском",
      "acceptance_criteria": [
        "Конкретный критерий приёмки 1",
        "Конкретный критерий приёмки 2"
      ],
      "test_steps": [
        "Шаг 1: действие",
        "Шаг 2: проверка",
        "Шаг 3: ожидаемый результат"
      ],
      "dependencies": [],
      "status": "pending",
      "assignee": null,
      "claimed_at": null,
      "lease_until": null,
      "review_required": true,
      "status_reason": null,
      "test_verdict": null,
      "test_owner": null,
      "test_started_at": null,
      "test_finished_at": null,
      "artifacts": []
    }
  ]
}
```

Обязательные поля каждой задачи:

- `id`
- `category`
- `priority`
- `description`
- `acceptance_criteria`
- `test_steps`
- `dependencies`
- `status`
- `assignee`
- `claimed_at`
- `lease_until`
- `review_required`
- `status_reason`
- `test_verdict`
- `test_owner`
- `test_started_at`
- `test_finished_at`
- `artifacts`

Начальные значения orchestration-полей для новых задач:

- `status`: `"pending"`
- `assignee`: `null`
- `claimed_at`: `null`
- `lease_until`: `null`
- `review_required`: `true`
- `status_reason`: `null`
- `test_verdict`: `null`
- `test_owner`: `null`
- `test_started_at`: `null`
- `test_finished_at`: `null`
- `artifacts`: `[]`

## Категории

Используй только эти категории:

- `infrastructure`: настройка проекта, зависимости, конфиги, CI, миграции, scaffolding.
- `functional`: бизнес-логика, доменные операции, core workflows.
- `ui`: интерфейс, UX-состояния, формы, навигация, пользовательские сообщения.
- `integration`: API, внешние сервисы, взаимодействие между подсистемами.
- `security`: авторизация, валидация, приватность, защита данных, secret handling.

Не создавай задачу в каждой категории механически. Категория нужна только если в PRD есть соответствующая работа.
Тесты и документацию включай в `acceptance_criteria` и `test_steps` соответствующих задач, если они нужны для завершения работы.

## Приоритеты

Используй только:

- `critical`: без задачи нельзя начать или проверить критический путь MVP.
- `high`: must-have для MVP, но не блокирует всю очередь.
- `medium`: важно для качества MVP, но можно выполнить после core path.
- `low`: polish, документация, улучшения или later-stage задачи.

Сначала должны идти инфраструктура и критический путь, затем функциональность, интеграции, безопасность, UI-polish, docs.

## Принципы Декомпозиции

### Атомарность

Одна задача должна быть выполнима одним агентом за одну рабочую сессию. Если задача выглядит больше чем на 30-60 минут или затрагивает несколько независимых областей, разбей её.

### Независимость

Минимизируй зависимости. Не делай длинную цепочку без необходимости. Если две задачи могут выполняться параллельно после одной базовой задачи, укажи одну общую зависимость, а не зависимость друг от друга.

### Проверяемость

Каждая задача должна иметь:

- 2-5 acceptance criteria;
- конкретные `test_steps`;
- хотя бы один способ проверки результата: unit/integration/e2e/manual.

Не пиши абстрактно: "проверить, что всё работает". Укажи действие и ожидаемый результат.

### Traceability

Каждая задача должна явно вытекать из PRD. Если требование в PRD неясное, сформулируй задачу через проверяемое поведение и добавь в acceptance criteria явное ограничение.

### Scope Control

Не добавляй фичи, которых нет в PRD, кроме минимально необходимых инфраструктурных, тестовых или security-задач. Если видишь важный пробел, не превращай его в скрытую фичу; добавь задачу на уточнение или зафиксируй вопрос в `docs/tasks/currentProblems.md`, если такой файл используется в проекте.

## Dependencies

Заполняй `dependencies` ID задач, которые должны быть завершены до старта текущей.

Правила:

- `TASK-001` обычно infrastructure/scaffolding и имеет `[]`.
- Не указывай зависимости только ради порядка.
- Не создавай циклические зависимости.
- Если задача зависит от нескольких foundation-задач, перечисли все реальные блокеры.
- UI-задачи обычно зависят от соответствующей функциональности или API-контракта.
- Integration/security задачи могут идти раньше UI, если они блокируют безопасный core flow.

После генерации мысленно проверь, что в очереди есть хотя бы одна ready-задача: `status = pending` и `dependencies = []`.

## Количество Задач

Ориентир:

- Маленький MVP: 8-15 задач.
- Средний MVP: 15-35 задач.
- Большой PRD: 35-50 задач.

Если получается больше 50 задач, сгруппируй PRD в этапы и создай задачи только для MVP/Phase 1, а later-stage работу вынеси в `docs/tasks/currentProblems.md` или отдельный backlog, если такой файл уже принят в проекте.

Если получается меньше 8 задач, проверь, не слишком ли крупные задачи и достаточно ли покрыты тестирование, security, интеграции и docs.

## Качество Acceptance Criteria

Пиши критерии приёмки как проверяемые утверждения.

Хорошо:

- "Пользователь с валидными credentials может войти и попадает на dashboard."
- "При невалидном файле пользователь видит понятное сообщение об ошибке, а обработка не падает."
- "Секреты интеграции читаются из переменных окружения и не логируются."

Плохо:

- "Сделать логин."
- "Улучшить UX."
- "Добавить безопасность."

## Качество Test Steps

Каждый `test_steps` должен быть исполнимым:

- укажи команду, если это автоматическая проверка;
- укажи пользовательский сценарий, если нужна ручная проверка;
- укажи ожидаемый результат.

Пример:

```json
[
  "Запустить uv run pytest tests/test_auth.py",
  "Создать пользователя с валидным email и паролем через UI или API",
  "Убедиться, что пользователь авторизован и видит dashboard без ошибок"
]
```

## Progress File

Если `docs/tasks/progress.md` отсутствует, создай его с минимальным заголовком:

```markdown
# Task Progress

Пока нет выполненных задач.
```

Если файл уже существует, не очищай его и не переписывай прошлые записи.

## Выполни

1. Прочитай обязательные входные файлы.
2. Извлеки из `docs/PRD.md` MVP scope, functional requirements, integrations, security/privacy requirements, UX requirements, milestones, assumptions и open questions.
3. Составь краткий внутренний план декомпозиции: foundation -> core flows -> integrations/security -> UI -> tests/docs.
4. Сгенерируй задачи с ID `TASK-001`, `TASK-002`, ... без пропусков.
5. Назначь категории, приоритеты и зависимости.
6. Проверь, что каждая must-have функция из PRD покрыта минимум одной задачей.
7. Проверь, что `docs/tasks/tasks.json` соответствует `docs/tasks/tasks.template.json` и является валидным JSON.
8. Создай `docs/tasks/progress.md`, только если его ещё нет.
9. В финальном ответе кратко укажи количество задач, диапазон ID, основные категории и любые открытые вопросы.
